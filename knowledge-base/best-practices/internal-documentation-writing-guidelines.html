Internal Documentation Writing Guidelines Published v1

Download original Word document

Prepared By: Angela Tino, Executive Operations Lead

Last Updated: July 22, 2026

Applies To: Anyone writing or editing a page in Canopi's Knowledge Repository

1. Purpose

As Canopi's repository grows and more people contribute to it, pages written by different people at different times should still feel like they belong to the same system. This guide sets out the conventions to follow so that stays true — not to add red tape, but so no one has to guess what a page should look like before they can write one.

2. Scope

Applies to anyone creating or editing a page in the repository — SOPs, policies, best practices, templates, or reference material of any kind.

3. General Principles

  • Write for someone who has no context — a new hire six months from now should be able to follow a page without asking someone to explain it first.

  • Be concrete. “The manager reviews it” is weaker than “Jagun Rimes reviews engineering PRs.” Name the actual person or role wherever possible.

  • Say what's unresolved instead of glossing over it. A clearly flagged open question is more useful than a page that quietly assumes an answer nobody's actually agreed on.

  • Keep pages evergreen — write so that updating one fact (a name, a tool, a step) doesn't require rewriting the whole page around it.

4. Document Types & Standard Sections

Two structures cover almost everything in the repository: SOPs (step-by-step processes) and policies (standing rules/expectations). Use the sections below as defaults, not a strict checklist — skip what genuinely doesn't apply.

Section Used In
Purpose Every document — SOPs and policies alike
Scope Every document
Team Reference SOPs where specific people/roles are involved in the process
RACI Matrix SOPs — who's Responsible, Accountable, Consulted, Informed at each step
Process Flow (Summary) SOPs — numbered, high-level steps
Detailed Steps SOPs — one subsection per step, each with Who / How / Output
Exceptions and Edge Cases SOPs — scenario / what-to-do table
Open Items to Finalize SOPs and policies with unresolved questions
Related Documents Every document — links to what it references or is referenced by

5. Formatting Conventions

  • Every page opens with the Canopi logo and a metadata block: Prepared By, Last Updated, and either Accountable Executive (SOPs) or Applies To (policies).

  • Use Canopi's brand purple (#2D005D) for headings, and Inter as the font throughout — consistent with Resources > Brand Assets.

  • Use numbered H1 sections for major topics, bullets for lists, and tables for anything comparing multiple items across the same categories (like a RACI matrix or a reference table).

  • Reference other pages by name rather than assuming the reader remembers where something lives — e.g. “see the Offboarding SOP” rather than “see the relevant process.”

6. Naming & Organization

  • File and page names should match the section titles used in the Knowledge Repository Content Plan (e.g. “Hiring & Recruitment SOP,” not “Hiring Process v2”).

  • Place each page under the nav section it belongs to per the Content Plan — if a new page doesn't obviously fit an existing section, that's worth flagging rather than force-fitting it somewhere.

  • Update the “Last Updated” date any time a page changes in a way that affects what someone reading it would do differently.

7. Tone & Voice

  • Plain language over formal-sounding language — write the way you'd explain it to a new teammate, not the way a legal contract reads.

  • Short sentences and direct instructions beat long, qualifier-heavy ones.

  • It's fine for a page to have personality — consistency doesn't mean every page has to sound robotic.

8. Flagging Open Items & Legal Review

Two different situations call for two different kinds of callout:

  • Use a light note (like this one) for an internal open question, assumption, or something to revisit — not urgent, just worth flagging.

Note: This is what an internal note looks like — use it for things like “confirm this with Jagun” or “this number is a placeholder.”

  • Use a bolder warning callout for anything that genuinely needs a professional (attorney, accountant, etc.) before the page should be relied on — reserve this for real legal/financial exposure, not routine uncertainty, so it doesn't lose meaning through overuse.

9. Keeping Pages Current

  • Review pages when the process they describe actually changes — don't wait for a scheduled review if something's gone stale in the meantime.

  • If a page is superseded by a new one, mark it clearly as outdated and link to its replacement, rather than deleting it outright or leaving two conflicting versions live.

  • Empty sections or “not yet formalized” notes are fine — they're placeholders, not failures. Better to have a visible gap than a confident-sounding page describing a process that doesn't actually exist yet.
