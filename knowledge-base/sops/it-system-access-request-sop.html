Purpose

This SOP documents how Canopi employees request access to internal tools and systems outside of standard new-hire provisioning, who approves and grants that access, and how access is tracked. This formalizes a process that has been handled informally to date.

Scope

Covers requests for access to any tool or system made after an employee's initial onboarding — e.g. a new tool the company adopts, a system a role expands into, or a one-off access need. Standard access granted as part of a new hire's setup is covered by the New Hire Onboarding & Equipment Setup SOP, and access removal is covered by the Offboarding SOP.

Note: An access tracker does not yet exist — this SOP introduces one. See “Access Tracker” below for what it should capture.

Process Flow (Summary)

  • 1. Employee identifies a tool/system they need access to

  • 2. Employee posts a request in the #access-requests Slack channel, naming the tool and reason for access

  • 3. Request is routed to the appropriate approver based on tool category (Bionka for company-wide/admin/financial, Jagun for engineering, Dixon for product/design)

  • 4. Approver reviews and approves (or follows up with questions) directly in the Slack thread

  • 5. Approver grants the access directly within the tool

  • 6. Angela Tino logs the granted access in the central Access Tracker

  • 7. Requester is notified in the Slack thread that access is ready to use

  • Detailed Steps
  • Step 1: Submit Access Request
  • Who: Requesting Employee

How: Post in the #access-requests Slack channel with the tool/system name and reason for the request. Flag clearly if the request is urgent.

Output: Request visible to the appropriate approver.

Step 2: Route & Approve Request

Who: Bionka Randolph, Jagun Rimes, or Dixon Lin, depending on tool category (see Tool Category Reference)

How: The approver for that tool category reviews the request and responds with approval or follow-up questions in the same Slack thread.

Output: Approved (or denied) request.

  • Step 3: Grant Access
  • Who: Same approver, unless delegated

How: Approver grants the access directly within the relevant tool or system.

Output: Employee has working access.

  • Step 4: Log Access
  • Who: Angela Tino

How: Angela records the request in the central Access Tracker — employee name, tool, access level, approver, and date granted.

Output: Up-to-date, centralized record of who has access to what.

  • Step 5: Confirm with Requester
  • Who: Approver or Angela Tino

How: Requester is notified in the Slack thread that access has been granted and is ready to use.

Output: Closed-loop request.

Access Tracker

Since no tracking exists today, this SOP introduces a simple log to be maintained going forward (a natural fit for Tools & Systems > System Documentation in the repository). Recommended columns:

  • Employee name

  • Tool/system

  • Access level (e.g. viewer, editor, admin)

  • Approved by

  • Date granted

  • Date revoked (filled in during offboarding or when access is no longer needed)

Exceptions and Edge Cases

Scenario What to Do
Tool doesn't clearly fit an existing category (e.g. brand-new tool being adopted for the first time) Default to Bionka Randolph as approver until the tool is added to the Tool Category Reference table with a clear owner.
Tool involves billing, customer data, or financial systems None currently in use. If/when adopted, require Bionka Randolph's approval regardless of category, given the added sensitivity.
Request is urgent / time-sensitive Requester flags this clearly in the Slack request. Approver aims to respond same-day where possible.
Primary approver is unavailable Bionka Randolph can act as a fallback approver for any tool category.
